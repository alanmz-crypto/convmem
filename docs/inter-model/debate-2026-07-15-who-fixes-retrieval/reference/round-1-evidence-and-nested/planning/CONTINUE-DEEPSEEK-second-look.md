## Flag: dedupe direction reversal (subtle behavior change)

**What the architecture says (step 3):**

> Drop from **recent** any unit whose `ledger_id` already appears in semantic (semantic keeps content; recent inject is novel-only).

**What the current code does (ask.py ~179-186):**

```python
recent_ids = { ... from recent }
rest = []
for unit in semantic:
    lid = ... unit.metadata.ledger_id ...
    if lid and lid in recent_ids:
        continue          # ← DROP FROM SEMANTIC
    rest.append(unit)
```

The **current** behavior: when a ledger_id appears in both recent and semantic, the semantic hit is dropped and the recent decision survives. The design assumption is "recent decisions are authoritative — if the corpus has a cached version and we have an approved decision, prefer the approved decision."

The **proposed** behavior (step 3): when a ledger_id appears in both, the *recent* decision is dropped and the semantic hit survives. The design assumption is "the corpus already has this information — recent inject should only add novel content."

**Which is correct?** It depends on which is *richer*:

- The semantic hit carries the full decision text from the inter-model doc (parsed by the adapter with section structure). It has more context.
- The recent-injected unit carries a summary generated by `decision_record_to_unit` — the same factual content, compressed.

For the "purge-drift deferred" query both produce the same answer. For queries where the semantic hit is a partial section and the recent summary is a complete fact, losing the recent version could degrade the answer. But those cases are rare — the inter-model doc adapter parses entire sections, and recent summaries are derived from the same content.

**Recommendation:** This change is defensible but wasn't in my original corrections. Have Kiro confirm: is semantic-keeps-content the intended behavior, or should the architecture preserve the original dedupe direction (prefer recent)? If Kiro intended this reversal, the exact text is correct. If unintended, step 3 should be "drop from semantic any unit whose ledger_id already appears in recent" (current behavior).

**Not a blocker** — both directions produce functionally correct context blocks. The difference is whether a redundant recent decision wastes a recent budget slot (step 3 as written: recent is dropped, budget is freed for other recent decisions) or wastes a semantic slot (current: semantic is dropped, recent budget is consumed on redundancy).

## One minor mermaid clarification

The mermaid diagram shows:

```
Dedup[drop recent already in semantic]
Sem --> Merge[prepend then semantic]
```

`Sem` routes directly to `Merge`, skipping `Dedup`. This is correct — the dedupe only modifies recent decisions, not semantic. But the visual layout places the Dedup node between semantic and merge, which could mislead an implementer into thinking semantic passes through dedupe. Add a comment to the mermaid or reposition:

```
Sem --> Merge    %% semantic goes direct; dedupe only touches recent
```

Trivial — not a blocking issue.
