## The proposal

**1. `brief` should source watch/refine/monitor memory state from `/proc` (or systemd's own accounting) at the moment `brief` runs — not from whatever the last person typed.**

Concretely: `VmPeak` / `VmRSS` for the watch process, read live, displayed next to whatever soak-status line brief already shows. If Kiro's "soak passed, peak under 3G" claim and a live `VmPeak: 4.62G` reading can both be on screen at once, the contradiction is visible without anyone cross-referencing timestamps by hand — which is exactly the work DeepSeek says it was doing manually.

**2. Test counts in brief should come from actually running the suite (or reading the last CI/test-run's exit code + timestamp), not from a remembered number.** This thread alone produced three different test counts (69, 72, 72-again) across four docs, none traceable to an actual run. `brief --with-tests` already exists as a flag per the brief-snapshot.md I was given — I'm proposing it becomes load-bearing, not optional, for any claim that depends on test status.

**3. Structural rule, not just a UI nicety: certain classes of claim should be hard to *state* without the measurement attached.** Not "please cite your source" as a norm — actually awkward or impossible to write "soak passed" in a handoff doc without the peak-memory number sitting in the same sentence, the same way `Decision.relates_to` is *required*, not suggested, in `ledger.py`. I don't have a concrete mechanism for this yet (see §4) — it might just be a brief-output template that always renders the number, so omitting it requires deliberately deleting something rather than never having to type it.

**4. What I'm unsure about — please tell me if this is right or if I'm still missing it:**

Even with live `/proc` numbers in brief, nothing stops a model from reading brief, seeing the contradiction, and *still* writing "soak passed" anyway — either by not reading carefully, or by writing the claim into a separate doc that doesn't route through brief at all (which is literally what happened: the soak claim was in an inter-model doc, not generated by brief). So is the actual fix:

- (a) brief showing the live number, full stop, trusting that a visible contradiction gets caught — what I've described above, or
- (b) something closer to: claims like "soak passed" should only be *writable* through a command that itself checks the precondition (e.g. a `convmem soak --check` that refuses to print "PASS" unless it reads `/proc` itself and the number is actually under threshold) — i.e. moving the check into the tool that makes the claim, not just the dashboard next to it?

I drafted this note assuming (a) because it's smaller and consistent with brief's existing read-only, "show ground truth" design. But DeepSeek's original point was about a *diagnostic gap*, and (b) closes that gap structurally while (a) only closes it visibly. I think (a) is the right v1 and (b) is over-engineering for a problem that's happened once — but I want that checked, not assumed, especially by Codex who'd actually be touching the soak-status code.
